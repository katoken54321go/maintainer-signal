# Codex for OSS Readiness Plan

This project can become a good Codex for OSS candidate only if it gains real usage and shows active maintenance. Do not apply with inflated claims.

## What the Official Form Asks For

- Public GitHub profile.
- Public GitHub repository URL.
- Maintainer role: primary maintainer or core maintainer.
- Why the repository qualifies, including stars, downloads, or ecosystem importance.
- How API credits would be used for the project.
- OpenAI organization ID.

## Honest Positioning

`maintainer-signal` helps OSS maintainers prepare truthful project-health evidence and spot missing maintenance hygiene. Its initial ecosystem value is strongest for small and medium projects that need a repeatable way to explain maintenance load, adoption signals, and responsible automation plans.

## 30-Day Plan Before Applying

1. Publish the repository publicly on GitHub.
2. Add a first release with a short demo report generated from this repository.
3. Use the tool on at least three real public repositories and record what it helped find.
4. Add one optional public-data collector, such as GitHub releases or npm downloads.
5. Open issues for roadmap items and close at least one pull request.
6. Ask a few maintainers to try it and collect public feedback.
7. Add screenshots or example reports to the README.

## Example Qualification Draft

`maintainer-signal` helps OSS maintainers collect truthful repo-health evidence: license, security policy, CI, tests, issue templates, release signals, and adoption notes. It is aimed at reducing repeated maintainer admin work and making project stewardship easier to review.

## Example API Credit Usage Draft

API credits would be used to build optional Codex-powered maintainer workflows: PR review summaries, issue triage drafts, release-note generation, documentation gap reports, and security-review assistance. All AI output would remain reviewable by maintainers before publication.

## Evidence To Gather

- GitHub stars and forks.
- Release count and release notes.
- Download metrics if published to npm.
- Public examples of generated reports.
- Issues triaged or pull requests reviewed with the tool.
- Maintainer feedback and external mentions.
